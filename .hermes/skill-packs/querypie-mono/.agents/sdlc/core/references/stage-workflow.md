# Stage Workflow

이 문서는 모든 SDLC 단계가 공통으로 따르는 내부 공정을 정의한다.

현재는 `sdlc-plan`만 구현되어 있지만, 같은 구조는 `sdlc-design`, `sdlc-build`,
`sdlc-test`, `sdlc-review`, `sdlc-documentation`, `sdlc-release`에도 적용한다.

## Four-Step Loop

모든 단계는 다음 네 공정을 가진다.

1. 입력
2. 티키타카
3. 출력
4. 출력 보정

이 네 공정은 사용자에게 모두 드러날 필요는 없다. 사용자는 자연어로 요청하고,
Agent는 내부적으로 이 공정을 따라 단계 산출물을 만든다.

## 1. 입력

입력은 단계가 판단에 사용할 공식 재료를 모으는 공정이다.

입력은 세 종류로 나눈다.

- 사용자 입력: 아이디어, 질문, 결정, 우선순위, 일정, 제약
- 외부 입력: Jira, GitHub issue, Slack thread, 고객 제보, 운영 기록
- 내부 입력: `.sdlc` case 문서, `docs/` 문서, source code, 이전 단계 handoff

Agent는 입력을 사람이 모두 복사해 주기를 기다리지 않는다. 접근 가능한 source는
직접 조회하고, 확인한 근거를 산출물에 남긴다.

입력 공정의 결과는 “무엇을 알고 있고, 무엇을 모르는가”를 분리한 상태여야 한다.

## 2. 티키타카

티키타카는 Agent와 사용자가 함께 판단을 다듬는 공정이다.

Agent는 질문을 한꺼번에 많이 던지지 않는다. 판단을 진행하는 데 필요한 질문부터
짧게 묻는다.

필요하면 역할 관점 검토를 기본 검토로 수행한다. 사용자가 별도 역할 Agent 검토를
승인하면 native agent를 시도하고, 지원되지 않으면 동일 세션에서 역할별 검토를 수행한다.

- 제품 관점
- 기술 관점
- UX 관점
- QA 관점
- 배포와 운영 관점
- 위험 검토 관점

Agent는 선택지를 제안하고, 사람은 방향, 범위, 우선순위, 진행 여부를 결정한다.

티키타카 공정의 결과는 “결정된 것”과 “아직 열린 것”이 구분된 상태여야 한다.

## 3. 출력

출력은 단계의 판단을 공식 문서로 남기는 공정이다.

출력은 대화 내용을 그대로 붙이는 것이 아니다. 다음 단계가 대화 기록 없이 읽고
이어갈 수 있도록 구조화해야 한다.

모든 단계 출력은 다음 성격을 가져야 한다.

- 현재 단계의 결론
- 결론의 근거
- 확정 결정
- 열린 결정
- 제외 범위
- 다음 단계가 반드시 읽을 문서
- 다음 단계가 하면 안 되는 일
- 다음 단계 완료 조건

출력 공정의 결과는 `.sdlc/cases/<case-id>/` 아래의 stage 문서로 남긴다.

## 4. 출력 보정

출력 보정은 산출물을 다음 단계로 넘기기 전에 읽기 쉽고 이어받기 가능한 상태로
만드는 공정이다.

출력 보정은 다음을 포함한다.

- 문서 구조 검증
- 문서 품질 검증
- 빈 placeholder 제거
- 현재 단계 책임의 누락 확인
- 다음 단계 handoff 확인
- 사람이 승인해야 할 우려 분리

사용자는 보통 `마무리해줘`라고만 말한다. Agent는 내부적으로 `stage-finish.md`를
따라 보정과 검토를 수행한다.

출력 보정 공정의 결과는 `마무리 완료`, `보완 필요`, `진행 승인 필요` 중 하나여야
한다.

## Short Feedback Loop

뒤 단계에서 앞 단계의 결정이 틀렸거나 빠졌다는 사실이 드러날 수 있다.

이때 Agent는 완료된 앞 단계 문서를 조용히 수정하지 않는다. 먼저 사용자에게 되돌릴
단계와 닫아야 할 질문 1개를 설명하고, 승인되면
`.agents/sdlc/core/references/stage-backtrack.md`를 따른다.

Backtrack은 전체 case를 처음부터 다시 시작하는 절차가 아니다. 필요한 앞 단계만
`draft`로 다시 열고, downstream 산출물은 stale 상태로 표시한 뒤 재확인한다.

## Stage-Specific Differences

네 공정의 틀은 같지만 단계마다 내용은 다르다.

`sdlc-plan`:

- 입력: 아이디어, 요청, 외부 신호, 기존 제품 문맥
- 티키타카: 문제, 목표, 범위, 기대효과, 진행 추천을 다듬는다.
- 출력: planning result와 design handoff를 작성한다. 설계 후보는 남기되 확정하지 않는다.
- 출력 보정: 계획이 설계 단계로 넘어갈 만큼 충분한지 확인한다.

`sdlc-design`:

- 입력: plan 결과, 확정된 제품 방향, 제약, 관련 코드 구조
- 티키타카: 설계 선택지, tradeoff, build 단위를 다듬는다.
- 출력: design result, build tasks, build handoff를 작성한다.
- 출력 보정: 구현자가 대화 기록 없이 작업을 시작할 수 있는지 확인한다.

`sdlc-build`:

- 입력: design 결과, build tasks, 관련 source code
- 티키타카: 구현 순서, 위험, 의존성을 다듬는다.
- 출력: 구현 결과와 남은 작업을 기록한다.
- 출력 보정: test 단계가 검증할 수 있도록 변경 내용과 위험을 정리한다.

`sdlc-test`:

- 입력: build 결과, 테스트 대상, 품질 위험
- 티키타카: 검증 범위, 자동/수동 테스트, 남은 위험을 다듬는다.
- 출력: 검증 결과와 남은 품질 위험을 기록한다.
- 출력 보정: review 단계가 판단할 수 있도록 증거와 실패를 정리한다.

## Rules

- 대화 기록은 공식 상태가 아니다.
- 중요한 결정은 checkpoint나 stage 문서에 남긴다.
- 앞 단계 수정이 필요하면 backtrack 절차를 사용한다.
- Agent가 추천하고 사람은 결정한다.
- 출력은 한국어로 작성한다.
- 코드는 planning 단계에서 수정하지 않는다.
- 출력 보정 없이 단계를 완료했다고 말하지 않는다.
