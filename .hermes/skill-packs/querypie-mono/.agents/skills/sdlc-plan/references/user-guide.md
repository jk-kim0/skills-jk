# SDLC Plan User Guide

이 문서는 사용자가 `sdlc-plan`을 어떻게 요청하고 마무리하는지 설명한다.

사용자는 내부 script, 검사 이름, 상태값을 외울 필요가 없다. 자연어로 요청하면
Agent가 필요한 수집, 대화, 문서 작성, 마무리 점검을 진행한다.

## 시작하기

계획을 시작하려면 `/sdlc-plan`이나 `sdlc-plan으로 계획 세워줘`처럼 요청한다.

예시는 다음과 같다.

- `/sdlc-plan QPD-1234 계획 세워줘`
- `요즘 사용자가 불만을 많이 말하는 부분을 찾아서 계획 후보를 제안해줘`
- `이 아이디어가 할 만한지 Jira와 기존 문서를 보고 같이 판단해줘`

## 준비 요약 확인

Agent는 바로 공식 case를 만들지 않는다.

먼저 Jira, GitHub issue, Slack thread, 기존 문서, 코드 근거처럼 확인 가능한 자료를
찾고 `계획 준비 요약`을 보여준다.

준비 요약에는 다음 항목이 들어간다.

- 만들 case 후보
- case가 하나의 trunk-based PR로 review 가능한 범위인지에 대한 판단
- 확인한 입력 자료
- 부족한 자료
- 사용할 검토 방식
- 공식 case를 만들지 여부에 대한 승인 요청

자료가 부족하면 Agent는 한 번에 많은 질문을 하지 않고, 진행에 가장 필요한 질문부터
하나씩 묻는다.

## Case 생성 승인

사용자가 준비 요약을 승인하면 `.sdlc/cases/<yyyy-mm-dd-name>/` 아래에 공식 case가
생긴다.

승인 전 자료 수집 결과는 `.agents/runs/` 아래 임시 공간에 둘 수 있다. 이 임시 자료는
git에 올리지 않고, 최종적으로 남길 가치가 있는 요약만 case 문서로 승격한다.

## 계획 대화

계획 단계는 brainstorm을 포함한다.

Agent는 외부 자료, 기존 문서, 코드의 영향 가능성, 역할별 관점을 바탕으로 질문하고
선택지를 제안한다.

사용자는 방향, 범위, 우선순위, 진행 여부를 결정한다.

계획 단계는 case 경계도 판단한다. 승인된 case는 기본적으로 하나의 trunk-based PR로
review 가능한 범위여야 한다. 여러 PR이 필요해 보이면 Agent는 case split을 제안한다.

plan 단계는 설계를 확정하지 않는다. 기술 선택과 구현 구조는 다음 design 단계에서
결정한다.

## 마무리하기

계획이 충분하다고 판단되면 사용자는 `마무리해줘`라고 말하면 된다.

Agent는 내부적으로 문서 품질, 빈 항목, 다음 단계 handoff, 내용상 완료 가능 여부를
확인한다.

마무리 결과는 다음 셋 중 하나로 안내된다.

- `마무리 완료`: design 단계로 넘길 수 있다.
- `보완 필요`: Agent가 고칠 문서 문제나 사용자가 결정할 항목이 남아 있다.
- `진행 승인 필요`: 우려는 있지만 사용자의 판단으로 다음 단계 진행이 가능하다.

## 산출물 위치

승인된 plan 산출물은 다음 위치에 저장된다.

```text
.sdlc/cases/<yyyy-mm-dd-name>/
  README.md
  metadata.yaml
  evidence.md
  plan/result.md
  plan/handoff.md
```

현재 상태는 `metadata.yaml`에만 기록한다.

`README.md`와 `plan/result.md`는 상태값을 따로 복사하지 않고, 사람이 읽을 계획 내용과
다음 단계에 필요한 정보를 담는다.
