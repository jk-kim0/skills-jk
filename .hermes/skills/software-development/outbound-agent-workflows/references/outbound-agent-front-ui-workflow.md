# Outbound Agent Front UI Workflow

Use this skill when working in `querypie/outbound-agent` on front-end UI changes, especially `front/src/components`, App Shell, top-bar controls, dropdown menus, shared UI class tokens, or UI changes derived from `docs/ui/*` guidance.

## Workflow

1. Confirm repo state before editing.
   - Run `git status --short --branch` in the repo root.
   - Fetch/update latest `origin/main` and use a repo-local `.worktrees/<topic>` worktree for file changes.
   - Do not modify the root checkout except for read-only inspection or fast-forwarding clean local `main` when appropriate.

2. Ground the UI change in canonical docs and existing implementation.
   - Read relevant `docs/ui/*` docs first, such as `docs/ui/display-size-guidelines.md`, `docs/ui/screen-overview.md`, and `docs/ui/terms.md`.
   - Inspect existing shell/control code before adding new UI: `front/src/components/app-shell.tsx`, route-local components, `front/src/lib/ui/class-names.ts`, and any design-contract tests.
   - When updating UI design docs for an already-implemented page, name page regions/parts by the actual Named Component names in the current implementation (for example `PageHeader`, `EmailSenderRegistrySection`, `EmailSenderEntityCardList`, `EmailSenderEntityCard`) rather than generic phrases like “영역”, “부분”, “card list”, or “wrapper section”. Add a short component-name map when the doc introduces the screen.
   - If a prior PR is referenced, verify whether it is merged and inspect the landed files on latest `main` rather than relying on the PR title/body alone.

3. Preserve top-bar ordering and future reserved slots.
   - When adding or moving a top-bar utility near Help, place it intentionally relative to the build version and Help.
   - Current outbound-agent top-bar direction is to keep user-facing utility clutter folded under a single `Help` dropdown when the user asks to replace separate compact buttons.
   - For Help consolidation work, implement `version → Help Dropdown Menu → notification indicator` in `topbar-tools`; do not leave separate `FV`, `Hidden`, or direct `/docs` Help triggers next to it.
   - Structure the Help Dropdown Menu in the requested order when applicable: `User Manual` linking to `/docs`, separator, `Hidden pages` section with the hidden page list, separator, then `Feature Visibility` section with the three visibility options.
   - If latest `main` already contains `HelpMenu`, do not leave implementation plans saying “convert Help to a dropdown” or “create `help-menu.tsx`.” Rebase first, inspect `front/src/components/app-shell.tsx` and `front/src/components/help-menu.tsx`, then phrase follow-up work as modifying the existing `HelpMenu` and adding any new section in the current menu order.
   - Preserve Feature Visibility state/localStorage behavior by moving only the option UI into the Help menu; a state helper module may still own `visibilityOptions`, storage key, event name, subscribe/read helpers, and `shouldShowFeatureStatus`.
   - When Feature Visibility affects Sidebar navigation, make the Sidebar itself subscribe to the same helper (`useSyncExternalStore` + `subscribeFeatureVisibilityChange`) instead of treating the Help menu as only a label/control surface. Assign explicit feature statuses to each sidebar entry and filter with `shouldShowFeatureStatus` so `Released only`, `Released + in progress`, and `All features` change actual visible navigation.
   - If hidden/mock review routes are requested in Sidebar only for `All features`, reuse the canonical `hiddenPages` list from `AppShell`, pass it into `NavLinks`, and render it as subordinate links directly under the relevant parent entry (for example `Settings`) only when `visibilityMode === "all"`. Keep the Help menu hidden-pages list if it remains part of the agreed dropdown contract, and update source tests to assert both locations deliberately.
   - When the user asks for Feature Visibility to “work” on Sidebar/App Shell menus, wire the menu component itself to the shared helper rather than just changing Help menu labels: make the client menu subscribe with `useSyncExternalStore(subscribeFeatureVisibilityChange, readStoredFeatureVisibility, readDefaultFeatureVisibility)`, add an explicit `status: FeatureStatus` to each menu item, and filter with `shouldShowFeatureStatus(item.status, visibilityMode)`. For the current outbound-agent Sidebar contract, `Released` items are `Home`, `Campaigns`, `Assets`, `Email Senders`; `Settings` is `In-Progress`; `Work Queue` is `Mock` and should only appear under `All features`.
   - If the user says a utility should be “right next to Version” or that a version label will be placed to the left of the new button, implement the requested source order in `topbar-tools`, not inside the Team selector/context group.
   - When a top-bar utility dropdown is moved or consolidated, update structure/design tests that pin the source order, absence of old triggers, section ordering, and option counts so future edits do not drift it back into `topbar-context` or separate buttons.

4. Prefer a focused client component for browser-side controls.
   - Put browser-only interactions such as `window.resizeTo`, `window.innerWidth`, localStorage, or `useSyncExternalStore` into a separate `"use client"` component.
   - Keep `AppShell` as a thin server component that imports and places the client control.
   - Define standard option lists as exported constants when tests or future docs may need to verify them.
  - For debug-only visual aids such as component-name inspection, prefer a single global client overlay plus explicit DOM markers (`data-component-name`) over making every component render its own hover label. The component should declare its code-matching React component name; the overlay should own pointer tracking, label positioning, localStorage state, and non-interference (`pointer-events: none`, `aria-hidden`).
  - For component-name inspection controls, prefer a four-mode selector rather than a binary toggle when the user needs focused inspection, ancestor context, and full-screen structure review: `Off`, `Pointer`, `Pointer + Ancestors`, and `Always`. Prefer `Pointer` over `Hover` because the implementation tracks pointer targets, not CSS `:hover`; prefer `Pointer + Ancestors` over `Hover & Ancestors` for clearer menu wording. Put these under a distinct `Show Component Name` section in the Help dropdown, use radio-style menu items, and persist the mode (not a boolean). Apply shell-wide: authenticated `AppShell` uses the existing `HelpMenu`, while public `DocsShell` needs an equivalent docs-header debug menu because it does not have the authenticated Help dropdown. For broad marker coverage, plan a stacked PR sequence and defer `Always` until route-local marker coverage is in place; early PRs should implement only `Off`, `Pointer`, and `Pointer + Ancestors`. If a keyboard shortcut is requested, document a low-conflict cycling shortcut such as `Alt+Shift+N` / `⌥⇧N`, avoid Chrome DevTools picker conflicts like `Cmd/Ctrl+Shift+C`, and ignore the shortcut while `input`, `textarea`, `select`, or `contenteditable` has focus. When adding click-to-copy for displayed component-name labels, make only the label copy target interactive: keep the overlay root non-interfering with `pointer-events: none`, make the label button `pointer-events: auto` with `cursor-copy`, call `navigator.clipboard.writeText(componentName)`, and preserve the previous pointer target while the pointer is over the label so the overlay does not immediately disappear before the click. Update `DESIGN.md`, `docs/feature/show-component-name-debug-toggle.md`, and the relevant OpenSpec scenario together, especially removing stale “copy-to-clipboard excluded” or “label pointer-events none / aria-hidden overlay” wording. See `references/component-name-debug-mode-stacked-plan.md` for the staged implementation pattern.
  - If a requested top-bar Help utility needs controls under “Help” but `Help` is currently a plain `/docs` link, plan or implement a `HelpMenu` client component that preserves the existing Docs navigation as a menu item and adds the new control below it, rather than adding a separate adjacent top-bar button without addressing the requested menu location.
  - For desktop-only sidebar collapse/rail controls, separate raw state from effective state. Use a viewport-gated `effectiveCollapsed = isDesktop && collapsed` (for example via `useSyncExternalStore` + `matchMedia`) so mobile width keeps expanded single-column semantics even if the user collapsed the sidebar on desktop before resizing. Hide the collapse toggle on mobile (`max-[900px]:hidden` or equivalent), keep Team switcher/footer/nav labels rendered on mobile, and update docs/PR body/tests to state that collapse is desktop-only.

5. Use shared class tokens and structure tests.
   - Add or reuse tokens in `front/src/lib/ui/class-names.ts` instead of scattering one-off class strings through the component.
   - When extracting a wrapper/client component around existing shell regions, audit any Tailwind arbitrary child selectors such as `[&>nav]:...`, `[&>section]:...`, or route-local direct-child selectors. If the DOM gains an intermediate wrapper, move the affected layout/scroll styles to an explicit wrapper class token (for example `sidebar-navigation`) and add a structure test that rejects the stale direct-child selector.
   - When AppShell content-container guidance or OpenSpec says Hidden mock-up routes must use `AppShellContent` as the only route-level Content Container, do not satisfy Component Name Debug coverage by adding `Hidden*Content` page-local wrappers with `mx-auto`, `max-w-*`, `min-h-full`, `bg-*`, or `p-*` classes. Remove those wrappers from every locale authoring file for the hidden route, keep markers on the header and Named Section Components, and update tests to assert the forbidden wrapper/marker is absent (`not.toContain`) rather than present. If a forbidden wrapper appears in sibling hidden pages named in docs/spec examples (for example `HiddenEntityCardExamplesContent`), fix those in the same follow-up so the new rule is not contradicted by another hidden review route.
   - Update design-contract tests such as `front/src/components/app-shell-design.test.ts` when placement/order is part of the product expectation.
   - For simple UI contracts, string-based Vitest tests can verify labels, standard sizes, and shell placement without starting a dev server.
   - When a UI PR intentionally removes or shortens explanatory copy, update source-level contract tests to encode the new absence/placement contract instead of resurrecting the removed copy just to satisfy stale assertions. For example, if repeated Email Sender reuse guidance is removed from both Sales Person creation and Team Email Sender Settings, assert the long guidance strings are absent from both sources and keep the remaining dropdown/action contract assertions focused on behavior.
   - When a user reports padding around a text/button footer area, distinguish text typography from container spacing before editing. Inspect the underlying shared primitive defaults as well as the local class: shadcn `CardFooter` currently includes `"flex items-center px-6 [.border-t]:pt-6"`, so a local `border-t` footer may still get 24px top padding even if the component adds `py-*`. Prefer overriding the actual container top padding (for example `pt-2 [.border-t]:pt-2` with an intentional `pb-*`) and keep the helper text `text-*`/`leading-*` unchanged unless the user explicitly asks to change the text itself. Add a source test for the primitive-default override when this is the root cause.
   - When replacing visible text brand marks with SVG/Image-based Logo Brand components, update the accessible label source as part of the same change. In Docs Shell this may live in locale page `docsShellLabels.brand`, not in the shell component itself; add a source-level test that rejects stale `Outbound Agent ...` brand labels when the visible brand is `Outbound`.
   - When moving an existing App Shell control between regions (for example Top Bar → Left Sidebar), assert both the new order and the absence/staleness of the old placement expectation in the design contract test.
   - For submit actions inside Radix/shadcn `DropdownMenuItem asChild`, avoid nesting the repo `Button` primitive as the child. Use the established form submit pattern from `WorkspaceSwitcher`: `<form action={...}>` → `<DropdownMenuItem asChild>` → native `<button className={uiClass("menu-button")} type="submit">...`. This keeps the item visible and preserves the menu item class contract. Add a source-level AppShell/design test that rejects the stale `<DropdownMenuItem asChild className={uiClass("menu-button")}>` + `<Button type="submit" variant="ghost">` shape for account-menu actions such as `Sign out`.
   - For simple Account Dropdown links inside Radix/shadcn `DropdownMenuItem asChild`, explicitly apply the repo typography token on the item wrapper, for example `<DropdownMenuItem asChild className={uiClass("menu-item")}>` → `<Link ...>...`. Do not leave account links as bare `<DropdownMenuItem asChild>` children, because Radix/shadcn defaults can make link item typography drift from the repo menu tone while submit actions use `menu-button`. When reviewing whether Account Settings and Log out/Sign out match, do not stop at source token presence; verify browser computed styles for link vs submit items (`font-size`, `font-weight`, `line-height`, `height`, `border-radius`, padding, hover/focus background) because Tailwind/shadcn cascade can leave a native button rendered as `16px/400/38px` while the link is `13px/700/34px`.
   - When a dropdown component-name debug marker must identify a menu whose panel renders through a Radix portal, mark both the visible trigger/owner wrapper and the `DropdownMenuContent` portal panel with the same component name. For Account Dropdown work, prefer extracting a small `AccountDropdownMenu` component and applying `componentNameDebugProps("AccountDropdownMenu")` to both places; otherwise the trigger may label correctly while the opened menu panel does not show the component name. Treat Component Name Debug label overlap as debug-only evidence as long as the component names remain identifiable; do not classify label overlap as a blocker. Review normal dropdown behavior with Component Name Debug Off as the primary contract.
   - For Account Dropdown menu structure, keep the closed trigger concise: show only the account display name in the trigger. When the opened panel needs user identity context, render it as the first non-action summary row in `account name` newline `<email>` order, then render the focused actions below it. If the current request only asks for Account Settings and Sign Out below the identity row, do not preserve older admin/system entries such as `System Settings` merely because they existed in the previous implementation; update source-level tests to assert the exact menu order and absence of stale entries.
   - When source-level tests assert dropdown placement or menu structure, avoid overfitting to one-line JSX strings. If the implementation becomes multiline to add debug props or other attributes, assert stable fragments such as `className={uiClass("menu-panel menu-panel-top")}` and `side="top"` separately.
   - When Sidebar item visibility depends on Feature Visibility, update both source-level design-contract tests and browser-facing review specs: assert the default `Released only` menu count/labels, the `All features` expanded labels, and any subordinate hidden/mock links under their parent. Also update `DESIGN.md`, `docs/ui/screen-overview.md`, `docs/ui/sidebar-navigation-ia-plan.md`, and `docs/feature-status.md` so docs do not still describe static always-visible Sidebar entries.
   - When rebasing a Sidebar IA/navigation PR and `front/src/components/nav-links.tsx` or `nav-links.test.ts` conflicts, preserve the latest `main` entrypoints and icons, then add the PR-specific entrypoint(s) in the documented order. Update the string-based nav tests together: expected `label` order, `buildTeamHref(...)` assertions, released/mock/in-progress status buckets, and relative order checks should reflect the merged IA rather than choosing one side of the conflict.

6. For settings/profile UI refactors, preserve the data contract deliberately.
   - If the user asks to remove a broad edit section but keep one editable property, prefer a dedicated server action/service for that single property instead of reusing a multi-field update action with hidden inputs for all required fields.
   - Hidden required fields can fail when current data is optional or not set, and they can accidentally keep unrelated fields editable through the back door. A single-field action should validate only the field being changed and revalidate the same route/layout paths as the broader action.
   - When the user says rarely changed settings do not need an aggregate editor, keep them as read-only rows with small per-row `Edit` triggers. Each row Dialog should save only that row's field(s): for example Team name, Team slug, and Team market should use separate actions/services instead of resurrecting an `Edit Team` section.
   - For inline Vercel-style settings cards such as `StringSettingCard`, compare helper/footer spacing against the user's reference screenshot and avoid oversized helper-text gutters. If the footer helper line looks padded or vertically heavy, prefer compact footer tokens (`min-h` around 52px, `py-2`, smaller helper text/line-height) while preserving button alignment, and add a source-level test that rejects the previous oversized footer contract.
   - For grouped-but-atomic rows such as Team market, edit the natural row unit together (`country` + `language`) while still avoiding unrelated Team fields such as name, slug, or profile image.
   - Reuse existing primitives such as `front/src/components/ui/dialog.tsx` for pop-up/Dialog edits before adding a new modal implementation.
   - In route-local app pages, keep the page readable by extracting meaningful local components such as `TeamProfileSection`, `TeamProfileImageChangeDialog`, `TeamNameEditDialog`, `TeamSlugEditDialog`, `TeamMarketEditDialog`, and `TeamDangerZoneSection` rather than leaving long anonymous sections or moving copy into a generic renderer.
   - For provider-backed settings entities created through OAuth, use an always-visible provider-specific Optional Create card when the provider identity is only known after callback. Do not ask for the email/identity before OAuth; create after callback with the verified provider identity, reject duplicates against existing non-deleted entities in the same Team/scope, and expose Delete-only controls if Disconnect/reconnect is not a product behavior. Review hard-delete vs soft-delete feasibility against entity-model references before implementing. See `references/provider-oauth-create-card-pattern.md`.
   - When latest `main` has Entity Card state-variant contracts, map provider Optional Create cards to the optional-provider-addition meaning, not just to a plus-action visual. If the provider can be added even when existing entities already exist, use `optional-configuration` / `optional-configuration` content presence rather than `required-creation` / `creation-required`; reserve `required-creation` for setup that blocks progress until the entity exists. If the shared optional card component lacks an action slot, extend the shared widget and add source tests that the provider card does not regress to required-creation.
   - For Optional Create / provider-addition Entity Cards, treat the centered circular `+` as the primary CTA itself. Do not add a separate bottom blue text/button CTA unless the current UI spec explicitly asks for it. Disabled optional-create states should keep the same centered `+` affordance in a disabled style rather than replacing it with a text button. When changing this contract, update the shared widget tests, route-local provider card tests, UI design docs, and OpenSpec SHALL/SHALL NOT scenarios together.
   - When provider-specific Optional Create cards already communicate the available action, avoid broad top-of-section helper/guidance copy unless the current UI spec explicitly requires it. If the user asks to remove such guidance, remove the rendered block and unused component, drop redundant headings, add source tests that assert the stale helper copy is absent, and update route-specific UI docs, OpenSpec, and the PR `UI 변경` text together. See `references/provider-settings-guidance-copy-removal.md`.
   - For shared Entity Card state variants, do not assume the local Button primitive already provides pointer affordance. If the centered `+` is a CTA, render it as a real button/link action target, add explicit `cursor-pointer` (and disabled cursor handling when applicable), expose route/action props such as `optionalActionHref` or `creationActionHref` rather than leaving a decorative span, and add source tests that assert the CTA label/href/cursor contract. Hidden mock-up pages should include visible copy stating that Optional Create and Required Create centered `+` symbols are CTA buttons, pointer hover must show the pointer cursor, and click/keyboard activation must run the connected setup/create action.
   - For normal shared Entity Cards, keep secondary actions such as `Open`, `Edit`, `Preview`, `Detail`, or route-provided `card.action` out of the title/name row. Prefer a card-local hover/focus action overlay (`position: absolute`, `group-hover`/`group-focus-within`, `pointer-events-none` until visible) so the default card footprint emphasizes entity type/status/name/meta/preview and the CTA layer does not reserve a grid column or push the title area. Update `docs/ui/entity-card-widget.md` and add source-level tests that assert `data-entity-card-cta-layer="hover-overlay"`, hidden-by-default opacity, hover/focus visibility classes, and absence of the stale `grid-cols-[minmax(0,1fr)_auto]` title-row layout for both standard and compact cards.

7. Keep UI design docs in sync with shell placement and entity-boundary changes.
   - For App Shell placement changes, update `docs/ui/screen-overview.md` and the relevant shared widget doc such as `docs/ui/common-widgets.md` in the same PR.
   - If the change affects Sidebar IA expectations, also update `docs/ui/sidebar-navigation-ia-plan.md` so docs no longer point users to the old menu location.
   - Replace old location language explicitly (for example “Top Bar Team selector”) rather than only adding new copy, otherwise design docs will contradict the implementation.
   - When Team Company/Product/Sales Person relation changes affect UI, mirror the canonical model boundary instead of turning all prerequisites into form blockers. For outbound-agent's current Team-rooted model, Team Company is a Campaign readiness/submission requirement and belongs in Assets/Campaign readiness surfaces when missing; Product and Sales Person creation forms should remain available because those records are Team-scoped reusable assets, not Company children.
   - Add or update source-level tests that reject stale hard-gate copy such as `Create a Team Company before adding Products/Sales People`, while preserving Campaign create/readiness checks for the Team Company context.

7A. For docs-only UI design/OpenSpec requests, inspect implementation but do not implement code unless explicitly asked.
   - When the user asks to improve a route's UI design and says to update 설계 문서/OpenSpec, treat it as documentation/spec work by default.
   - Read the live route implementation enough to identify current section names, copy location, wrapper surfaces, and data/order assumptions, but keep file changes in `docs/**` and `openspec/**`.
   - If no route-specific UI doc exists, create a focused `docs/ui/<route-or-surface>.md`, add it to `docs/ui/README.md`, and connect it from the relevant feature doc.
   - For `/{teamSlug}/assets` docs-only design changes, keep the route-specific doc and cross-links synchronized across `docs/ui/assets.md`, `docs/ui/README.md`, `docs/ui/screen-overview.md`, `docs/ui/uri-map.md`, and `docs/ui/sidebar-navigation-ia-plan.md` when IA wording is affected. See `references/assets-setup-card-section-design.md` for the Required Create / Optional Create / Current assets section pattern.
   - For iterative UI-design sessions where the user gives step-by-step label/copy/layout decisions, keep a canonical route-specific `docs/ui/<route-or-surface>.md` decision log with a current implementation Component Name Map, accepted decisions, rationale, and implementation impact; update that document on each instruction before or alongside code changes.
   - If the user's later instruction changes backend/API behavior for the same UI flow (for example what entity is created when a button is clicked), stop treating the task as docs-only: record the PO decision in the relevant feature/backend design doc and implement the service/test changes needed to make the contract true.
   - When a UI-design iteration turns a setup card for a reusable Team asset into a Campaign-scoped choice/new flow, implement the full repeated pattern for each affected asset class rather than only the first card: setup card action → `/{teamSlug}/campaign/{slug}/<asset>-choose`, existing Entity Card selection action, required-creation card → `/{teamSlug}/campaign/{slug}/<asset>-new`, create-and-assign server action, route-local locale pages, source-level route/action tests, UI doc decision log, and OpenSpec route contract. For outbound-agent Simple Email Sending this pattern applies to Contact List, Message, and Sales Person; Message assignment uses the latest Email Template Version on the draft SendRun, while Sales Person assignment updates Campaign `salesPersonId` and the draft SendRun sender identity.
   - Record accepted product choices in the active OpenSpec `design.md` plus SHALL/SHALL NOT scenarios in the relevant spec, including explicit non-goals such as “UI does not invent client-side default sorting.”
   - For Campaign-scoped asset choice/create flows where a Team asset is assigned into a Campaign child entity, use the pattern in `references/campaign-scoped-entity-choice-flow.md`: preserve `/{teamSlug}` routes, add `/{teamSlug}/campaign/{slug}/...` choice/new screens, reuse the domain assignment service in server actions, and update slug/id read-model support plus docs/OpenSpec together.
   - When a user changes a generated Campaign slug rule during a UI-flow task, treat it as a product/domain contract, not just a helper tweak. Update the canonical feature/UI decision docs, implementation constants, deterministic generator tests, and a collision retry test that proves the candidate is checked in the same scope as the schema constraint before use. For Simple Email Sending, the accepted pattern is 4-character lowercase alphanumeric (`/^[a-z0-9]{4}$/`) generated slug, checked against existing Campaign slugs in the current Team (`Campaign @@unique([teamId, slug])`) and retried on collision.
   - For Entity Card-style absence states, distinguish passive empty/no-item states from required-creation cards; required setup should be documented as a required-creation card, not generic empty text.
   - When the user introduces a provisional Entity Card Variant name but says its meaning will be clarified in a separate PR/document, keep the current PR narrow: record only the default usage rule and the fact that the detailed meaning/visual rules are deferred. Do not rename or rewrite every related UI document, visual-design doc, or route-specific card rule unless the user explicitly asks for that broader terminology migration.
   - For root-level system/admin surfaces such as `/system/`, explicitly distinguish them from Team-scoped settings (`/{teamSlug}/settings/**`) in the feature/UI plan. Document the route-level access guard, where the entrypoint appears, and the visible paths that need PR `UI 변경` coverage.
   - For root-level system/admin surfaces such as `/system/`, explicitly distinguish them from Team-scoped settings (`/{teamSlug}/settings/**`) in the feature/UI plan. Document the route-level access guard, where the entrypoint appears, and the visible paths that need PR `UI 변경` coverage.
   - When a feature plan asks whether a rarely changed setting needs a DB table or code/YAML configuration, include a concrete persistence trade-off in `Domain / Data / API 영향`: code/YAML + env vars, existing model reuse, and new DB table options; do not default to a settings table unless runtime editing/audit is in scope.

8. Verification should be narrow unless the user requests local full verification.
   - Link worktree `front/node_modules` to the root checkout when needed: `cd front && ln -s ../../../front/node_modules node_modules`.
   - Run `git diff --check`.
   - Run focused Vitest tests for changed components or services.
   - If focused Vitest fails at import time because `@/generated/prisma/client` is missing in a fresh worktree, run `npm run prisma:generate` from `front/` (use Node 24 when Prisma 7 is in play), then rerun the same focused test. Treat this as a generation prerequisite, not as a test failure to work around.
   - Run focused ESLint on changed files. With the current ESLint flat config, do not use Next-era `npm run lint -- --file ...`; run from `front/` with explicit path arguments instead, for example `./node_modules/.bin/eslint src/components/foo.tsx src/__tests__/foo.test.ts --max-warnings=0`.
   - If a front-end task touches `front/prisma/schema.prisma`, run `npm run prisma:validate` and `npm run typecheck` from `front/` using Node 24.
   - If a front-end task changes `next/link` href plumbing, shared component href props, or dynamic route strings passed into `Link`, run `npm run typecheck` from `front/` using Node 24 even when focused Vitest and ESLint pass. Next typed routes can fail only at production/typecheck time; type shared link props as `Route` from `next` and cast dynamic route construction at the caller boundary (for example ``const href = `/api/...?...` as Route``) rather than leaving shared props as plain `string`.
   - In outbound-agent, Prisma schema changes are folded into the single baseline migration `front/prisma/migrations/20260530000100_baseline_main_schema/migration.sql`; do not add a second migration directory unless the repo policy/tests change. The CI test `src/__tests__/schema-migration-artifacts.test.ts` currently rejects additional migration directories.
   - Do not start a dev server or run full build/test unless explicitly requested or necessary for confidence.

9. Commit, push, and create/update the PR.
   - Re-fetch and rebase onto latest `origin/main` before pushing.
   - After pushing, re-query the PR with `gh pr view <number> --json headRefOid,baseRefOid,mergeStateStatus,statusCheckRollup` and compare the PR `baseRefOid` with the freshly fetched `origin/main`. Also compare `headRefOid` with `git rev-parse HEAD` and, if GitHub still shows the pre-push head right after a successful push, verify `git ls-remote origin refs/heads/<branch>` and retry `gh pr view` after a short wait before interpreting CI/check state. If `main` advanced while the work was being rebased or pushed, fetch/rebase/push again before reporting that the PR is on the latest main.
   - Commit only relevant files.
   - Push the feature branch and create/update the PR with a Korean title/body for this repo.
   - For PRs that include UI changes, include a required `UI 변경` section in the PR description.
     - List URI paths where the changed UI can be checked as bullet items.
     - Use Demo Scenario team slugs in those paths when team-scoped routes are involved: `sales-demo`, `querypie-jp`, `querypie-kr`, and `querypie-us`.
     - Under each URI path, add nested bullets summarizing what changed on that screen in roughly 1–3 lines.
     - If 10 or more URI paths changed, list at most 10, prioritizing paths that show the major changes, and omit the rest.
   - When the user asks for a screenshot comment on a UI PR, first read the PR `UI 변경` section and the Vercel PR comment/status to get the exact Preview base URL. Capture the listed paths using full Before/After URLs: Before should be the stable main/dev deployment for the exact path, and After should be the PR Preview URL for the exact path. If the changed behavior is interactive (hover, focus, opened dropdown, selected mode), include an interaction-state screenshot in addition to the default state. If a listed route currently does not render the affected component or interaction target, still include the route screenshot when useful, but explicitly say in the PR comment that the route had no visible target for that interaction instead of implying the behavior was verified there.
   - For local screenshot upload comments, prefer `gh attach` for the image files, then write a normal PR comment that embeds the returned raw URLs. Include `Before:` and `After:` as full clickable URLs for each path, not only route fragments.
   - Use the repository/user GitHub CLI policy; on this workstation that commonly means `env -u GITHUB_TOKEN gh ...`.
   - When uploading PR screenshot evidence with the local `gh attach` extension and `--repo`/`-repo` is needed, pass the repo flag before the PR number using the extension's single-dash form, for example `env -u GITHUB_TOKEN gh attach -repo querypie/outbound-agent 337 path/to/screenshot.png`. If the local worktree branch name differs from the remote PR branch, do not rely on auto-detection; pass the PR number explicitly.
   - Check CI after PR creation/update. If checks are still running after a short active watch, report the in-progress checks rather than waiting silently for a long time.

10. Revert already-merged frontend PRs with a new PR.
   - Confirm the target PR is `MERGED` and read `mergeCommit.oid` with `gh pr view <pr> --json number,title,state,mergeCommit,mergedAt,url`.
   - Start from latest `origin/main` in a repo-local worktree and run `git revert -m 1 --no-edit <mergeCommitOid>`.
   - Verify the revert diff only cancels the target PR, run `git diff --check`, then push and create a Korean revert PR.
   - Do not include issue auto-close keywords in the PR body unless the user explicitly requests issue closure.
   - See `references/revert-merged-frontend-pr.md` for the command sequence and pitfalls.

## Pitfalls

- Do not implement from a PR body alone. PR # references may be stale or already merged; inspect the landed docs/code on latest `main`.
- Do not create separate `node_modules` in a worktree unless the root checkout dependencies are unavailable or incompatible.
- Do not hide browser-only APIs inside server components.
- Do not substitute full local builds for fast PR delivery when the user has asked to prefer commit/push and CI monitoring.
- Do not leave the user with only a local change; for repo work, push and PR reporting are part of completion unless explicitly excluded.
- When a PR body contains backticked paths or commands, write the body to a temporary file and use `gh pr create --body-file` or `gh pr edit --body-file`; do not pass a double-quoted multiline `--body` through the shell because command substitution can execute the backticked text.
- When rebasing a UI PR after a route-local i18n refactor has landed, do not resolve conflicts by resurrecting the old `page.tsx` body. Keep the latest thin route entry, move any PR-added visible sections/copy into each `page.en.tsx`, `page.ko.tsx`, and `page.ja.tsx` authoring module, and adjust source-structure tests to read the locale module rather than `page.tsx`.
- When rebasing a settings/profile UI PR after latest `main` extracted page sections into a wrapper component (for example `TeamSettingsManagementSections`), preserve the latest extracted wrapper and insert the PR-added card/section inside that wrapper. Do not flatten the page back to the older inline JSX just to keep the PR hunk.
- When rebasing App Shell/sidebar UI work after latest `main` changed the Logo Brand implementation, preserve the latest brand component/assets instead of resurrecting older text or initials. If collapsed sidebar needs a compact brand, prefer the latest symbol/logo variant from the current brand component rather than reintroducing stale `OA`/`Outbound Agent` text.
- When rebasing an App Shell/sidebar PR after latest `main` adds or moves component-name debug markers (for example `AppShellMainArea` / `AppShellContent`) while the PR extracts a client `AppShellFrame`, preserve the latest marker contract by passing debug props through the frame rather than dropping the markers or reverting to the old inline shell.
- If a shared widget gains variants that do not need normal entity actions, preserve the route-local copy boundary: normal entity cards should still require page-supplied labels, while actionless variants may render without those labels only when they truly have no drag/detail copy.
- For Entity Card state/add variants in outbound-agent, use the canonical names `Empty Card`, `Optional Add Card`, and `Required Add Card`. If updating code/contracts, keep names consistent across `EntityCardVariant`, `contentPresenceState`, internal component/function names, data attributes/tests, docs/OpenSpec, hidden example pages, and visual asset filenames. Avoid resurrecting older terms such as `State-only Card`, `Optional Configuration Card`, or `Required Creation Card`.
- When rebasing Email Sender settings PRs after Dry-Run Sender or Entity Card center-plus CTA work landed on `main`, preserve both contracts instead of choosing one side wholesale: Dry-Run rows remain non-Gmail/no-auth entities, Gmail OAuth starts from the Optional Create Card, existing Gmail sender rows stay Delete-only if that is the PR lifecycle, and action-connected optional cards use the `circle-action` centered `+` CTA contract. See `references/email-sender-rebase-conflict-pattern.md`.

## Example: standard viewport dropdown

For a top-bar dropdown that resizes the current browser viewport from `docs/ui/display-size-guidelines.md`:

- Use a separate client component like `front/src/components/viewport-size-menu.tsx`.
- Include the standard options from the doc: Desktop review `1440 x 1000`, Desktop smoke `1280 x 720`, Tablet portrait `834 x 1194`, Tablet landscape `1194 x 834`, Mobile display `393 x 852`, Mobile browser `393 x 659`.
- Show the current `window.innerWidth × window.innerHeight`.
- When resizing, compensate for browser chrome using `window.outerWidth - window.innerWidth` and `window.outerHeight - window.innerHeight`, then call `window.resizeTo(option.width + chromeWidth, option.height + chromeHeight)` from the user-triggered click handler.
- Mention in PR notes that normal browser windows may restrict `window.resizeTo` depending on browser policy/window state.
