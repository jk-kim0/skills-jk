# Gmail refresh-token-missing after dev DB reset

Use this reference when a deployed Outbound Agent dev environment redirects back to Email Senders with:

`gmail=failed&reason=gmail_refresh_token_missing&senderEmail=<email>`

## Interpretation

This error can be expected immediately after a dev database reset/seed if the selected Gmail sender row exists again but its `GmailSenderCredential` row was deleted by the reset.

The app needs one of two things to complete callback persistence:

1. a new `refresh_token` from Google; or
2. an existing stored Gmail credential for the selected Team + sender email + provider.

If Google omits `refresh_token` and the DB has no reusable credential, the callback fails with `gmail_refresh_token_missing`.

## Evidence pattern

Collect these signals before changing code or runtime config:

1. Confirm the OAuth start URL generated by the deployed app includes:
   - `redirect_uri=https://<env>/api/gmail/oauth/callback`
   - `access_type=offline`
   - `prompt=consent`
   - `scope` containing `https://www.googleapis.com/auth/gmail.send`
   - state created from the row-level `senderEmail`
2. Confirm the selected sender row exists but lacks a credential.
   - Browser-login to the dev app as the seeded demo user.
   - Call `/api/sender-identities` from the page context.
   - For the affected sender, check `status`, `emailAddress`, and whether `gmailCredential` is present; do not print encrypted token values.
3. Check recent environment DB workflows for reset+seed evidence:
   - `.github/scripts/tencent-vm.sh migrate --reset-data`
   - `prisma migrate reset --force`
   - `Database reset successful`
   - `prisma db seed`
   - `The seed command has been executed`
4. Confirm runtime version separately from public HTTP health.

## Common root cause

A user had already granted the Google OAuth client/scope before, so Google may omit a new refresh token. A dev DB reset then removed the app's previously stored credential. The callback can validate identity and selected sender successfully but still cannot persist credentials because no refresh token is available.

## Recovery

Tell the operator to revoke the app consent in the selected Google account's account access / third-party apps screen, then retry from the intended Email Sender row.

If the selected row/account is wrong, restart from the correct row instead. If the redirect reason is `gmail_alias_mismatch`, do not use this refresh-token recovery path; authenticate with the Google account matching the selected sender.

## Token exchange ambiguity

If the app redirects with `reason=gmail_refresh_token_missing`, do not assume it proves Google returned HTTP 200 with no `refresh_token`. Inspect the implementation and logs for whether token endpoint non-2xx responses are distinguished from a successful response that lacks `refresh_token`.

A robust callback should separate these cases:

1. Google token endpoint non-2xx, for example `invalid_client`, `invalid_grant`, or `redirect_uri_mismatch`:
   - classify as token exchange failure, not refresh-token-missing;
   - log only safe metadata such as HTTP status, provider error string, Team/sender identifiers, and redirect URI origin/path;
   - never log authorization code, access token, refresh token, client secret, cookies, encrypted token, or signed state payload.
2. Google token endpoint 200 OK but no `refresh_token`, and there is no existing credential:
   - classify as `gmail_refresh_token_missing`;
   - use the revoke-and-retry recovery path below.

When current logs do not include token response status/error, report the uncertainty explicitly: the observed failure proves credential persistence failed after callback, but it does not by itself prove whether Google omitted a refresh token or the token exchange failed and was over-broadly mapped to `gmail_refresh_token_missing`.

## Pitfalls

- Do not classify this as a redirect URI problem if the deployed OAuth start URL already shows the exact environment callback URI.
- Do not classify this as prompt/access_type missing if the URL already has `prompt=consent` and `access_type=offline`.
- Do not assume a row with `status=auth_required` has a credential; check `gmailCredential` explicitly.
- Do not overclaim the root cause from the user-facing `gmail_refresh_token_missing` reason if token endpoint non-2xx errors are mapped to the same reason in code.
- If direct SSH is unavailable from the agent workstation, use GitHub Actions logs and browser-authenticated app APIs for evidence instead of stopping the investigation.
