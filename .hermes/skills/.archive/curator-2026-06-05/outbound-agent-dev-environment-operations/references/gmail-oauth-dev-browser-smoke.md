# Gmail OAuth dev browser smoke

Use this when a dev environment reports Google/Gmail authentication failure but runtime env keys appear present.

## Why this smoke matters

Current app code derives the OAuth `redirect_uri` from the request public origin and forwarded headers, then appends `/api/gmail/oauth/callback`. A VM-local `GMAIL_OAUTH_REDIRECT_URI` can still be present and useful for operator sanity checks, but it is not sufficient evidence that the live OAuth start URL is correct. Verify the actual Google authorization redirect generated by `/api/gmail/oauth/connect`.

## Recommended evidence

1. Confirm latest deployed revision first.
2. Confirm runtime key presence without printing values:
   - `GMAIL_OAUTH_CLIENT_ID`
   - `GMAIL_OAUTH_CLIENT_SECRET`
   - `GMAIL_TOKEN_ENCRYPTION_SECRET`
   - `GMAIL_OAUTH_STATE_SECRET`
   - optional/operator sanity: `GMAIL_OAUTH_REDIRECT_URI`
3. If checking Tencent VMs, confirm file and container agree:
   - `/etc/outbound-agent/front.env`
   - `docker exec outbound-front printenv <KEY>`
4. Log in through the deployed site using a seeded reviewer account rather than hand-crafting a session cookie when possible.
5. Open a Team that actually has Gmail sender identities. The `sales-demo` personal Team may have zero sender identities after reset; use seeded demo Teams such as `querypie-jp`, `querypie-kr`, or `querypie-us` when validating Gmail sender OAuth links.
6. Follow a `Connect Gmail` link and inspect the resulting Google authorization URL:
   - host is `accounts.google.com`
   - `redirect_uri` equals the exact dev callback host, for example `https://outbound-seoul.dev.querypie.io/api/gmail/oauth/callback`
   - `scope` includes `https://www.googleapis.com/auth/gmail.send`
   - the URL/page does not contain `redirect_uri_mismatch`

## DNS/SNI workaround for operator machines

Some operator environments resolve Tencent dev domains to private `100.64.x.x` addresses. If public HTTPS or SSH checks become flaky because of that, force the public endpoint while preserving Host/SNI:

- curl: `--resolve outbound-seoul.dev.querypie.io:443:43.133.247.7`
- Playwright Chromium: `--host-resolver-rules=MAP outbound-seoul.dev.querypie.io 43.133.247.7`

Do not conclude the app is down solely from a private-IP TLS/SSH timeout if the GitHub Actions deploy/smoke or a public-IP/SNI-pinned request succeeds.

## Minimal Playwright shape

Run against the deployed URL, do not start a local dev server. After login:

1. `goto('/querypie-jp/settings/email-senders')`
2. collect `a[href*="/api/gmail/oauth/connect"]`
3. navigate to the first link
4. inspect `page.url()` for the Google authorization URL and query params above

Keep the report value-focused: deployed SHA, env key presence, connect link count/team used, generated `redirect_uri`, scope evidence, and whether `redirect_uri_mismatch` was absent.
